Phase Breakdown

What happens in each phase.

Every step has a purpose. Nothing is bureaucracy.

01

Boot — Session Alignment

Runs once at the start of every session. Takes about 30 seconds.
  • → Pull latest changes from git so the pearl library is current
  • → Detect project name, dev port, dev server directory, and memory path
  • → Start the dev server from the correct directory (not assumed to be root)
  • → Load only the Guidance tables from pearls.md — Hard Constraints first, then Heuristics, then Patterns
  • → Flag any pearls past their Review By date
  • → Activate ⚡ invocation tracking — triggers are armed
  • → Load session memory and CLAUDE.md project context
  • → Check harvest staleness — warn if overdue
  • → Report alignment status: pearls loaded, confidence breakdown, unvalued count
02

Active Session — Wisdom in Action

Runs continuously throughout the session. Invisible when working well.
  • → Before any substantial operation: briefly state which loaded pearls are relevant
  • → When a pearl's trigger condition is detected: consciously evaluate whether it applies
  • → When a pearl prevents a mistake: log ⚡ Pearl invoked: "[Title]" — [what was avoided]
  • → When a pearl was evaluated but didn't apply: note "Pearl '[Title]' considered — not applicable here"
  • → In long sessions (1+ hour): re-state Hard Constraint pearls before high-risk operations
  • → When problems arise: solve them normally — the timestamps are being recorded automatically
03

Harvest — Measure, Distill, Update

Runs at session end when significant work occurred. Takes 5–10 minutes.
  • → Scan all ⚡ invocation logs — increment Uses, recalculate Total Saved = Uses × Min/Use
  • → Retrospective scan: for each loaded pearl, did its trigger arise? Log unlogged opportunities
  • → Recalculate Hit Rate = Uses / Opportunities for every pearl
  • → For unvalued pearls (Min/Use = ?): ask user to estimate time cost, then backfill
  • → Promote maturity: Seed → Confirmed at 30 min saved, Confirmed → Established at 120 min saved
  • → Flag pearls past Review By date — user validates, extends, or retires
  • → Find problem cycles in the session: ATTEMPT → FAILURE → SOLUTION
  • → Measure elapsed time from timestamps — this is Min/Use for any new pearl
  • → Run 3-gate test: Non-obvious · Pain-tested · Transferable
  • → Check for patterns across pearls — propose meta-pearls if 3+ share a root
  • → Flag pruning candidates: Seed + Total Saved < 15 min + Age > 60 days
  • → Commit and push updated pearls.md
The Math

Two formulas that make the whole system work.

Simple, precise, and impossible to game. Value is earned, not assigned.

Pearl Value Formula
Total Saved =
Uses × Min/Use
Example: A pearl invoked 3 times that saves 120 min each time = 360 min saved
Why this matters: A pearl invoked once that prevents an 8-hour disaster scores 480. A pearl invoked 50 times saving 2 minutes each scores 100. Rare, high-impact pearls are 5× more valuable — and the system knows it.
Pearl Effectiveness Formula
Hit Rate =
Uses ÷ Opportunities
Example: Pearl fired 3 times out of 10 sessions where it should have fired = 30% Hit Rate
Why this matters: A pearl with Uses = 3 looks active. But if the trigger arose 30 times and it only fired 3 times, it's failing 90% of the time. Hit Rate reveals pearls that need better triggers or clearer wording — not pruning.
Where Min/Use Comes From
Min/Use =
Tfixed − Tproblem arose
Timestamps from the actual session. Not a guess — a measurement.
The key insight: The debugging session that creates a pearl already contains its value. Start the clock when the problem first appears. Stop it when the solution is confirmed. That elapsed time IS the exact cost — and therefore the exact future savings per invocation.

Pearl Lifecycle

Every pearl earns its place — or leaves with dignity.

Maturity is determined by measured value, not age or vote.

Seed
Total Saved: 0 → 29 min

Born from a real debugging cycle. Unproven. Has a Trigger and a measured Min/Use from day one. Loaded at boot, tracked carefully.

Confirmed
Total Saved: ≥ 30 min

Has prevented at least one real mistake since being written. Value is no longer theoretical. Promoted automatically when the threshold is crossed.

Established
Total Saved: ≥ 120 min

Proven across multiple invocations. This pearl has saved meaningful real-world time. Protected from pruning unless explicitly retired by a human.

Retired
Human confirmed outdated

Not deleted — archived. The reason it was retired is recorded. Institutional history of what was true and why it no longer is. Never auto-pruned.

Pruning rule: A pearl is flagged as a candidate only if all three conditions are true simultaneously: Maturity = Seed AND Total Saved < 15 min AND Age > 60 days. Pearls with unknown Min/Use are excluded from all pruning decisions. A human always makes the final call.

Pearl Types

Not all pearls work the same way.

Three types. Each measured differently. Each serving a different role.

Prevention

Prevention Pearls

Block a specific mistake at a specific decision point. The agent was about to do X wrong — the pearl surfaced and redirected. Logged as ⚡ invocations.

Example: "Always call Read before Edit" fires the moment the agent is about to call Edit on a file it hasn't read.

Measured by: ⚡ invocation logs → Uses × Min/Use = Total Saved
Orientation

Orientation Pearls

Frame the entire session's character and approach. Don't fire at a single decision point — shape every decision made throughout the work. The protection is atmospheric.

Example: "This is a template, not a project-specific tool" colors how every design decision is made, all session long.

Measured by: human judgment — did the agent need correction in this domain today?
Diagnostic

Diagnostic Pearls

Guide investigation when something breaks. Don't prevent the problem — accelerate the path to the solution. Fired when a known failure pattern is recognized.

Example: "If auto-research never completes, check PROCESS_TASK_URL first" cuts a 2-hour debug to 5 minutes.

Measured by: ⚡ invocation logs — time from problem onset to solution with vs. without the pearl
Why This Is Different

This has never been built quite like this before.

📐

Value Is Measured, Not Voted On

Min/Use comes from actual session timestamps. It is a measurement of real pain, not an opinion. No human bias. No inflated importance. The cost of the original mistake is the value of the pearl.

Rare Pearls Get Fair Credit

A pearl invoked once that prevents an 8-hour disaster is more valuable than one invoked daily for 30 seconds. Every existing system penalizes rare pearls. This one rewards them.

🎯

Hit Rate Finds Silent Failures

Tracking Opportunities (sessions where the trigger arose) alongside Uses exposes pearls that should be firing but aren't. Low Hit Rate is not a pruning signal — it is a tuning signal.

🌱

The Library Prunes Itself

Pearls that don't earn their place are flagged. Pearls that become obsolete are retired — never deleted — with their history preserved. The library stays lean and current automatically.

🔄

Compounding Returns

Each session improves the library. An improved library improves the next session. There is no ceiling on this accumulation. The value of the system grows with every hour of work.

Get Started

Four files. Zero infrastructure.
Running in under an hour.

1

Create your pearl file

Create .agent/alignment/pearls.md in your project root. Use the two-table format (Guidance + Metrics per category). Start with zero pearls — or seed it with a few obvious Hard Constraints you already know.

2

Add the /boot skill

Add boot.md to your Claude Code skills directory. Run /boot at the start of every session. It loads your pearls, arms your triggers, and reports alignment status in under 30 seconds.

3

Work normally. Log invocations.

When a pearl prevents a mistake, log it: ⚡ Pearl invoked: "[Title]" — [what was avoided]. That's it. The log is what harvest reads. No other changes to how you work.

4

Harvest at session end

Run /harvest after any session with significant problem-solving. It scans the conversation, measures time costs, extracts new pearls with real Min/Use values, updates all metrics, and commits. The library gets smarter.

5

Watch the library mature

After a few sessions, seeds become Confirmed. After a few more, they become Established. Hit Rate reveals which pearls need trigger refinement. Total Saved shows you exactly how much time the system has returned to you.
