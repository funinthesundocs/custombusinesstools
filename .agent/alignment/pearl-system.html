The Solution

A closed-loop memory engine that

The Pearl System gives AI agents something they have never had before: persistent, measured, self-pruning institutional memory. Built from real pain. Valued by real time saved.

PILLAR 01

Wisdom, Not Notes

Distilled universal rules, not raw notes. The 3-gate quality test ensures only pain-tested, non-obvious, transferable knowledge survives. A pearl that passes the gates is a pearl worth loading forever.

PILLAR 02

Value Is Measured

Every pearl has a price tag: Total Saved = Uses × Min/Use. A pearl invoked once that saves 8 hours (480 min) outranks one invoked 100 times saving 2 minutes each (200 min total). Frequency is not value.

PILLAR 03

The Loop Never Stops

Boot loads pearls → session applies them → harvest measures and distills → library matures. Each cycle makes the next one better. Compounding returns. The library is the proof.

Exact Command Sequence

The exact sequence. Every session.
No guesswork.

Three moments. Four commands. The whole system. Run these in order, every session, and the library takes care of itself.

1
Session Start
Run once. Every session. Takes ~30 seconds.
claude code cli
$ /boot

What it does:

  • Pulls latest changes from git (library is always current)
  • Reads .agent/alignment/pearls.md — loads all Guidance tables into active memory
  • Groups pearls by confidence: Hard Constraints first
  • Flags any pearls past their Review By date
  • Starts the dev server if not already running
  • Fires the boot canary: [BOOTED] N pearls loaded...
  • Invocation tracking ACTIVE — ⚡ logs enabled
2
During the Session
Log invocations as they happen.

When a pearl prevents a mistake or shapes a decision, the agent writes this exact format as text output in the Claude Code conversation:

agent output — written to the conversation
⚡ Pearl invoked: "Pearl Title" — I was about to [dangerous action]. Instead: [what the pearl says to do].
No special command required. The ⚡ line is just text output from the agent. At harvest, /wisdom-harvest scans the conversation for these lines to increment Uses counters and calculate Total Saved from the timestamps around them.
Retrospective Opportunity Scan: Even unlogged applications get captured. At harvest, the agent checks every pearl's Trigger condition against the full session — silent protections get credit too.
3
Session End
Commit your work first. Then harvest.
1

Commit your work

git
$ git add <your changed files> $ git commit -m "description of what you built"
Commit your actual work BEFORE harvest. These are separate commits with separate purposes. Harvest creates its own commit for pearls.md.
2

Run the harvest

claude code cli
$ /wisdom-harvest # or /harvest for shorter sessions

What /wisdom-harvest does automatically:

  • 01Scans session for ⚡ invocation logs → increments Uses → recalculates Total Saved
  • 02Runs Retrospective Opportunity Scan on all pearl triggers
  • 03Extracts problem cycles (attempt → failure → solution patterns)
  • 04Measures elapsed time from conversation timestamps (Min/Use = measurement, not estimate)
  • 05Runs 3-Gate Quality Test on every candidate — all three gates required:
    G1Non-obvious Would a capable practitioner get this wrong by default? Fail: any competent person already knows this.
    G2Pain-tested Did violating this cost measurable work time? Fail: fix took under 30 seconds. Auto-pass if measured time ≥ 30 min.
    G3Transferable Does this apply across 3+ meaningfully different contexts? Fail: only useful in one narrow scenario. Waived if measured time ≥ 4 hours.
  • 06Writes survivors to .agent/alignment/pearls.md with full two-table format
  • 07Promotes maturity: Seed → Confirmed → Established
  • 08git add .agent/alignment/pearls.md → git commit "harvest: N new pearls..." → git push
3

Done. Library updated.

Next session, /boot will load everything you just learned. The cycle continues.

When to use which harvest
/harvest

Quick harvest. Shorter sessions, few problem cycles. Same 19-step protocol, less ceremony. Use when the session was focused and you doubt new pearls emerged.

/wisdom-harvest

Full harvest. Deep sessions, multiple debugging cycles, new pearls likely. The comprehensive version. When in doubt, use this one — thoroughness is the point.

The Math

Four numbers that power
the whole system.

No votes, no subjective ratings. Every pearl's worth is a calculation — a measurement — not an opinion.

Pearl Value

Total Saved

Total Saved = Uses × Min/Use

The only metric that matters. A pearl invoked once saving 8 hours (480 min) is worth more than one invoked 100 times saving 2 minutes each (200 min). Frequency is not value.

Effectiveness

Hit Rate

Hit Rate = Uses / Opportunities

How often the pearl fires when its trigger arises. A low hit rate signals a trigger wording problem — the situation is happening, the pearl just isn't being applied.

Time Cost

Min/Use

Min/Use = T_fixed − T_problem_arose

Measured from conversation timestamps — the exact elapsed time of the debugging cycle this pearl prevents. A measurement, never an estimate. This is why the timestamps matter.

Library Quality

Health Score

0–100 composite

Maturity (30%) + Valuation coverage of invoked pearls (30%) + Avg hit rate (25%) + Prune cleanliness (15%). Shown live in the dashboard. A library at 80+ is a library that earns its load time.

Type

Prevention / Orientation / Diagnostic

Prevention blocks a mistake. Orientation shapes session framing before work starts. Diagnostic guides investigation when something breaks.

Trigger

The observable moment that forces evaluation

Vague triggers = low hit rate. "Before doing anything" is not a trigger. "Before calling Edit after a Grep search" is a trigger.

Confidence

Hard Constraint / Strong Heuristic / Soft Pattern

Hard = always follow. Strong = follow unless clear reason not to. Soft = background awareness, not a rule.

Opportunities

Sessions where the trigger condition arose

Counted whether or not the pearl fired. Used to calculate Hit Rate. The denominator of the effectiveness score.

Min/Use

Elapsed time of the original debugging cycle

Measured from conversation timestamps: T_fixed − T_problem_arose. A measurement, never an estimate. The exact cost of the mistake this pearl prevents.

Review By

Expiration date — 6 months from creation

Flagged at boot when past due. A human validates whether the rule is still accurate. Technology changes. Pearls should too.

Pearl Lifecycle

Every pearl earns its place —
or leaves with dignity.

Pearls are not permanent by default. They start as seeds and prove their value through invocations. Or they get flagged for review and retire gracefully — preserved as institutional history.

Seed
newly added

Add date set. Min/Use measured from the incident that spawned the pearl. Waiting to prove value across sessions.

Confirmed
≥ 30 min saved

Total Saved crossed 30 minutes. Pattern confirmed across sessions — this pearl fires when it should and saves meaningful time.

Established
≥ 120 min saved

Core library. Loads at every boot. Has saved more than two hours of work. Unlikely to be pruned.

Retired
archived, not deleted

Preserved as institutional history with retirement reason. Removed from boot load. Never deleted — the record of why we once needed this rule matters.

✂ Pruning Rule

Seed + Total Saved < 15 min + Age > 60 days = Prune Candidate. Automatically flagged for human review — never auto-deleted. You decide. The system only surfaces the candidates.

Pearl Types

Three ways a pearl protects a session.

Not all pearls do the same job. Type determines when a pearl fires and how its value gets measured.

Prevention

Blocks a specific mistake

Fires when a dangerous action is about to be taken. The trigger is a specific, observable decision point. Measurable: time saved = the debugging cycle that would have followed.

Measurement: T_fixed − T_problem_arose · Easiest to value precisely
Orientation

Shapes session framing

Prevents wrong architectural assumptions before work begins. Fires early in the session during planning. Harder to measure — value is in what doesn't get built wrong.

Measurement: time-boxed estimate of wrong-path cost · Conservative by convention
Diagnostic

Guides investigation

Fires when specific symptoms appear. Points investigation toward the likely cause, away from expensive wrong paths. Value = time saved by not following the wrong thread.

Measurement: elimination of known dead-end paths · Avg wrong-path time saved

Vision

What this becomes
over time.

The Pearl System is not a static configuration file. It is a living library that grows with every session. Each harvest adds pearls, updates metrics, and promotes the library's maturity — and the next boot starts from that improved state. The improvement is not linear. Each pearl reduces the frequency of expensive mistakes, which frees cognitive bandwidth for discovering the next category of patterns. The library deepens as it grows.

The dashboard shows this happening in real time. The Health Score rises as pearls move from Seed to Established. The Total Saved column fills in as invocations accumulate. The hit rate stabilizes as trigger wording improves. These are not vanity metrics — they are evidence of an agent that is measurably better at its job than it was six weeks ago.
